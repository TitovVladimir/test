# Менеджер задач

Интеллектуальный менеджер задач с канбан-доской и интеграцией LLM (ZhipuAI GLM) для категоризации, декомпозиции, приоритизации и аналитики.

## Реализованные функции

**Управление задачами:**
- CRUD-операции с задачами (создание, редактирование, удаление)
- Канбан-доска с тремя колонками: Ожидает, В работе, Готово
- Drag-and-drop перетаскивание карточек между колонками (@dnd-kit)
- Фильтрация по приоритету, сроку и текстовому поиску
- Система подзадач (декомпозиция)

**AI-функции (LLM):**
- Категоризация — автоматически определяет категорию задачи (Работа, Личное, Учёба и т.д.)
- Декомпозиция — разбивает задачу на 2-5 подзадач с возможностью редактирования
- Приоритизация — оценивает приоритет (LOW/MEDIUM/HIGH) с обоснованием
- Сводка — генерирует аналитический отчёт по нагрузке в Markdown

**Прочее:**
- Аутентификация по паролю с JWT-токеном (7 дней)
- Тёмная/светлая тема
- Адаптивный интерфейс (мобильные табы / десктопные колонки)

## Запуск

### Через Docker (рекомендуется для продакшена)

```bash
docker compose up -d --build
```

Поднимутся три сервиса: PostgreSQL (порт 5432), API (порт 3001), Web (порт 3000). При первом запуске будет применена схема базы данных через Prisma.

### Локальная разработка

1. Запустите PostgreSQL (локально или через Docker):
```bash
docker compose up postgres -d
```

2. Примените схему БД и сгенерируйте Prisma-клиент:
```bash
pnpm db:push
pnpm --filter api db:generate
```

3. Запустите API и фронтенд в режиме разработки:
```bash
pnpm dev
```

API будет доступен на `http://localhost:3001`, фронтенд — на `http://localhost:3000`.

### Сборка для продакшена

```bash
pnpm build
```

## Архитектурные решения

**Монорепо (pnpm workspaces + Turborepo).** Пакет `@task-manager/shared` содержит типы и Zod-схемы, общие для фронтенда и бэкенда. Это исключает расхождение контрактов между клиентом и сервером.

**Оптимистичные обновления.** Drag-and-drop на канбан-доске применяет изменения мгновенно в локальном стейте, а затем отправляет запрос на сервер. При ошибке — откатывается с уведомлением.

**Единый SortableContext на колонку.** Каждая колонка канбан-доски имеет свой `SortableContext` (@dnd-kit). При перетаскивании карточки между колонками отображается визуальный плейсхолдер. Такой подход避免了 дублирования sortable ID в двух контекстах одновременно.

**Прерываемые LLM-запросы.** Мутации `useSummarize` и `useDecompose` принимают `AbortSignal`. При закрытии модального окна запрос прерывается через `AbortController.abort()`, а состояние мутации сбрасывается через `reset()`.

**Prisma как ORM + валидация.** На уровне БД работает Prisma с PostgreSQL. На уровне API-контроллеров — class-validator DTO. На уровне фронтенда — Zod-схемы из shared-пакета.

**NestJS-архитектура.** Контроллеры → Сервисы → Репозитории (Prisma). Разделение ответственности по слоям. Глобальные `ValidationPipe` (whitelist), `HttpExceptionFilter` и `TransformInterceptor`.

## Известные ограничения

1. **Простая аутентификация.** Используется один пароль для всех пользователей. Нет регистрации, ролей и хеширования пароля (хранится в открытом виде в .env).

2. **Карточки не сдвигаются в целевой колонке при drag-and-drop.** Техническое ограничение @dnd-kit — нельзя иметь один sortable ID в двух `SortableContext` одновременно без побочных эффектов. При перетаскивании показывается пунктирный плейсхолдер.

3. **Нет полноценной миграции БД.** Используется `prisma db push` вместо `prisma migrate`, что не подходит для продакшена с существующими данными.

4. **LLM-ответы не кэшируются.** Каждый запрос к AI-функциям создаёт новый вызов к API.

5. **Нет пагинации на фронтенде.** Все задачи загружаются разом. При большом количестве задач может быть проблема с производительностью.

6. **Нет HTTPS.** SSL-терминация не настроена — предполагается использование reverse proxy (Nginx, Caddy) перед приложением.

## Что добавил бы при наличии времени

1. **Регистрация и авторизация пользователей** — полноценная система с ролями (админ, пользователь), хешированием паролей (bcrypt), refresh-токенами.

2. **Real-time обновления** — WebSocket (через NestJS Gateway) для синхронизации канбан-доски между несколькими пользователями.

3. **Уведомления о дедлайнах** — email/Push-уведомления о приближающихся и просроченных задачах через cron-задачи.

4. **История изменений** — аудит-лог действий с задачами (кто, когда, что изменил).

5. **Полноценные миграции Prisma** — замена `db push` на `migrate deploy` для безопасного обновления схемы в продакшене.

6. **Кэширование LLM-ответов** — хранение результатов категоризации/приоритизации в Redis, чтобы не вызывать API повторно для той же задачи.

7. **Пагинация на фронтенде** — виртуальный скроллинг или бесконечная подгрузка для работы с большим количеством задач.

8. **Тёмная тема для email/уведомлений** — системные уведомления с учётом пользовательских предпочтений темы.

9. **Мобильное перетаскивание** — улучшение drag-and-drop для тач-устройств (свайпы между колонками).

10. **CI/CD пайплайн** — GitHub Actions для автоматического тестирования, линтинга, сборки Docker-образов и деплоя.
